## Feature Brief
- Problem: The current 311 search is useful, but it is locked to one fixed radius and one fixed time window. That makes it hard to tune the results for different addresses or to move through larger result sets when the first page is not enough.
- Audience: Curious residents, renters, homebuyers, and neighborhood scouts who want to adjust how broad or recent their 311 search is without leaving the app.
- Why now: The app already proves the core address-to-nearby-complaints flow. The next useful step is to make the search criteria adjustable and let users move through more than one page of results so they can explore the neighborhood signal instead of only seeing a single fixed slice.
- Proposed direction: Add user-facing controls for search radius and time window, keep them constrained to a small set of presets for the first pass, and add explicit next/previous page controls for result browsing. Preserve the current default experience as a sensible starting point, but let users widen, narrow, or advance the search without rebuilding the flow.
- Constraints: Keep the first version mobile-first and simple. No auth. No backend. No saved searches. No map-specific paging. Prefer presets over freeform inputs if that keeps the experience faster and less error-prone. Continue using the existing Chicago 311 client and query layer.
- Non-goals: No advanced analytics. No arbitrary sort options. No infinite scroll. No multi-filter combinatorics. No geospatial drawing tools. No server-side caching layer.
- Success criteria: A user can change radius and time window from the app and immediately understand the effect on the result set. A user can move to the next or previous page without losing the current criteria. The app clearly shows when paging is available and keeps the current criteria visible enough that the search remains understandable.
- Risks or open questions: Bigger pages may increase request cost or latency, so the first pass should keep page size modest. The best preset values may need a couple of real-device passes. If the API’s paging semantics are awkward, we may need to keep paging client-managed rather than trying to make the server do too much.
- Recommended next step: Build the smallest adjustable search form with preset radius/time-window options and explicit next/previous controls, backed by paged client queries and a visible summary of the active criteria.
